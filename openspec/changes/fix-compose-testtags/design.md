## Context

В composeApp (`commonMain`) сегодня используется смешанный стиль `testTag`:

- `TaskCard.kt:38` — `testTag = "task_card_${taskList.id}"`
- `TaskItem.kt:40` — `testTag = "checkbox_${task.id}"`
- `CreateListScreen.kt:115` — `testTag = "list_name_field"` (ок)
- `CreateListScreen.kt:153` — `testTag = "task_input_$index"`

ID (`taskList.id`, `task.id`) — внутренние идентификаторы, генерируемые в репозиториях (`MemoryTaskListRepository`/бэкенд), тест заранее их не знает и не контролирует. Индекс `$index` тоже не является «пользовательскими данными» и меняется, когда пользователь удаляет/добавляет поля.

Автотесты на Assertum MCP находят элементы по `accessibility_tree` и запрашивают `testTag` вместе с `text`/`contentDescription`. Стабильный контракт для них — «фиксированный тег роли + пользовательские данные для дизамбигуации».

## Goals / Non-Goals

**Goals:**
- Убрать из `testTag` всё, что не контролируется пользователем (entity ID, порядковые индексы, которые меняются при редактировании).
- Зафиксировать правило: `testTag` описывает **роль** элемента UI, а не его экземпляр. Экземпляр определяется пользовательскими данными через `contentDescription` (для элементов без собственного `Text`) или через уже видимый текст (для элементов с текстом).
- Сохранить возможность писать стабильные Assertum‑сценарии вида «найти `task_card` с именем `Morning Routine`» без знания ID.
- Дать каждому `LazyColumn` собственный контейнерный `testTag`, чтобы тест мог скоупить поиск item‑ов до конкретного экрана.

**Non-Goals:**
- Не переделывать модель данных (`Task.id`, `TaskList.id` остаются).
- Не трогать web‑приложение, backend, репозитории.
- Не писать/не переписывать сами e2e‑тесты — только контракт в UI.
- Не вводить централизованный реестр констант тегов в этом change (тегов пока ~7 — оставим строковые литералы по месту).
- Не различать секции **Pending** и **Completed** внутри `ProgressScreen`: они живут в одном `LazyColumn`, и обёртывание в `Column` с собственным тегом сломало бы lazy‑рендеринг. В этом change мы вешаем только общий `progress_column`; дизамбигуация одноимённых задач между секциями остаётся на следующий change (см. Open Questions).

## Decisions

### Decision 1: `testTag` = роль, уникальность — через пользовательские данные

Принято: назначать один и тот же `testTag` всем экземплярам одной роли (`"task_card"`, `"task_checkbox"`, `"task_input"`). Отличать конкретный элемент в тесте — по `text`/`contentDescription`, собранным из пользовательских полей (`TaskList.name`, `TaskList.date`, `Task.name`, введённый пользователем текст).

Почему так, а не через ID:
- ID назначается репозиторием после сохранения — тесту (и UI‑прекондишену) этот ID на момент подготовки сценария ещё не известен.
- Имя списка/имя задачи тест задаёт сам — они детерминированы и читабельны в отчётах.

Альтернативы, которые рассмотрены и отклонены:
- **Оставить `testTag` с ID** — отклонено: тест не может предсказать значение.
- **Использовать тег с именем (`"task_card_${taskList.name}"`)** — отклонено: имя может содержать пробелы/спецсимволы, тег перестаёт быть «именем роли», и появляется сложность с неуникальностью имён. Нужные данные уже лежат в `contentDescription`/`Text`, их достаточно.
- **Использовать индекс (`"task_input_$index"`)** — отклонено по той же причине: индекс не является пользовательскими данными и меняется.

### Decision 2: Способ прокидки пользовательских данных

Для каждого места — минимально инвазивный способ:

- **`TaskCard`** — уже содержит `Text` с `taskList.name` и `taskList.date`. Для ясности в accessibility‑дереве добавляем `contentDescription = "${taskList.name}, ${taskList.date}"` на корневом `Column` карточки. Это не ломает screen reader и дополняет видимый текст.
- **`TaskItem` Checkbox** — у чекбокса нет собственного текста, поэтому `contentDescription = task.name` на модификаторе чекбокса. Тест делает `onNodeWithTag("task_checkbox") + hasContentDescription(task.name)`.
- **`CreateListScreen` task input** — поле ввода: `testTag = "task_input"`, а текущее значение поля (`taskText`) и так попадает в семантику через `OutlinedTextField.value`. Для пустого поля (placeholder «Type a new task…») добавим `contentDescription = "Task input"` как роль — дизамбигуация между несколькими пустыми полями ложится на порядок в accessibility‑дереве, это приемлемо, т. к. пустое поле — это всегда «последнее» в сценарии.
- **`CreateListScreen` list name field** — `"list_name_field"` уже стабилен. Оставляем.

### Decision 3: Контейнерный `testTag` на `LazyColumn`

Каждому `LazyColumn` в `commonMain` назначаем собственный стабильный `testTag` — «имя скоупа»:

- `MyListsScreen` → `"my_lists_column"`
- `CreateListScreen` → `"create_list_column"`
- `ProgressScreen` → `"progress_column"`

Имена — snake_case с суффиксом `_column`, чтобы однозначно отличить контейнер от item‑тегов (`task_card`, `task_checkbox`, `task_input`) в accessibility‑дереве и в отчётах тестов.

Зачем это нужно:

- **Скоуп поиска**. Тест может сказать «внутри `my_lists_column` найти `task_card` с `contentDescription = "Morning Routine, …"`» и не зависеть от других экранов/оверлеев, где мог бы появиться элемент с тем же тегом роли.
- **Якорь для list‑level ассертов**. Проверки «список не пуст», «список содержит N карточек», «скролл в начало списка» удобно адресовать к контейнеру, а не обходить все item‑ы.
- **Детерминированное поведение при навигации**. При переходе между экранами на экране может оказаться несколько LazyColumn одновременно (например, во время transition). Явный контейнерный тег снимает неоднозначность.

Ленивая композиция `LazyColumn` при этом сохраняется — `testTag` висит на самом `LazyColumn`, а не на невидимых детях; children продолжают композироваться по мере скролла.

Альтернативы, которые рассмотрены:

- **Не давать контейнерам тегов, полагаться только на уникальность `contentDescription`** — отклонено: работает, пока экран один, но ломается при наложении экранов/оверлеев и требует идеальной уникальности cd.
- **Один общий тег `"list"` для всех трёх LazyColumn** — отклонено: не снимает неоднозначность при транзитах, и разговор «в каком списке искать» приходится вести через наличие соседних элементов.

### Decision 4: Секции Pending/Completed в `ProgressScreen` — вне скоупа этого change

На `ProgressScreen` Pending и Completed находятся в **одном** `LazyColumn` как последовательные `items(...)`‑блоки с текстовыми заголовками между ними. Контейнерный `progress_column` даёт скоуп экрана, но не различает секции: задача `"Exercise"` в Pending и `"Exercise"` в Completed после этого change остаются неотличимы по паре `(testTag, contentDescription)`.

В этом change мы сознательно **не** решаем эту проблему. Обсуждённые варианты и причина откладывания:

- **Обернуть секции в `Column(testTag = "progress_pending_section")`** — даёт чистый скоуп, но каждая секция становится одним композируемым узлом → lazy‑рендеринг внутри секций пропадает. Для списков ≤ 20 задач на день это приемлемо, для сотен — нет. Решение о приемлемой границе — отдельный разговор.
- **Расширить `contentDescription` в `TaskItem`** до `"${task.name}, Pending"` / `", Completed"` — дёшево, но дублирует состояние чекбокса (`Checkbox.checked` и так в семантике) и делает экспорт cd громоздким для screen reader.
- **Опираться на `Checkbox.checked`** — тест может отличить секцию по состоянию, если уже нашёл элемент по имени. Работает, но только для чекбокса, а не, скажем, для кнопки edit.

Мы фиксируем выбор в Non‑Goals и в Open Questions — следующий change добавит дизамбигуацию секций, если такая потребность появится в реальных тестах.

### Decision 5: Где жить правилу

Правило фиксируется как ADDED‑требование в `mobile-ui-tests`, а не в `mobile-screens`. Причина: это контракт **для тестируемости**, а не для бизнес‑поведения экранов. `mobile-screens` описывает, что отображается пользователю; `mobile-ui-tests` описывает, как это отображение можно автоматизированно проверить.

## Risks / Trade-offs

- **[Риск] Существующие тесты сломаются** → Сейчас в `mobile-ui-tests` нет e2e‑сценариев, опирающихся на `"task_card_<id>"`/`"checkbox_<id>"` (видно по `spec.md`). Существующий Assertum e2e находит элементы по тексту и кнопкам, не по ID‑тегам. Регрессий не ожидается.
- **[Трейд‑офф] Несколько чекбоксов с одинаковым именем задачи в одном списке** → Модель `Task` допускает, но UI не запрещает. Если две задачи с одинаковым именем — `contentDescription` на чекбоксе дублируется. Приемлемо: тест в таком сценарии и сам должен задавать уникальные имена; а UX этот дубликат тоже делает неотличимым для зрячего пользователя.
- **[Трейд‑офф] `contentDescription` озвучивается screen reader'ом** → это добавляет в озвучку `"Morning Routine, Sep 23"` для карточки. Считаем это плюсом для a11y, а не минусом.
- **[Риск] Дрейф правила при добавлении новых компонентов** → Без централизованного реестра тегов новые компоненты могут снова получить ID‑теги. Митигирует требование в `mobile-ui-tests` — оно становится чек‑пунктом при ревью и при написании будущих e2e.

## Migration Plan

1. Обновить пять файлов: `TaskCard.kt`, `TaskItem.kt`, `CreateListScreen.kt` (item‑теги + `testTag` на LazyColumn экрана создания), `MyListsScreen.kt` (`testTag` на LazyColumn), `ProgressScreen.kt` (`testTag` на LazyColumn).
2. Собрать `:composeApp:assembleDebug` — убедиться, что компиляция и Compose‑семантика проходят.
3. Запустить `:composeApp:allTests` — быстрый smoke, т. к. unit‑тестов на эти компоненты сейчас нет, проверка нужна только на отсутствие падения компиляции в `commonTest`.
4. Прогнать существующий Assertum e2e — убедиться, что он по‑прежнему зелёный (он не опирается на удаляемые ID‑теги).

Откат: revert коммита — изменения локальные, без миграции данных.

## Open Questions

- Нужно ли выносить строки тегов в общий `object TestTags` в `commonMain`? → На текущем объёме (~7 тегов) — нет. Возвращаемся к этому, когда появится 10+ компонентов с `testTag` или первый дубликат тега.
- Когда понадобится различать секции Pending и Completed внутри `ProgressScreen` — делаем это отдельным change. Триггер — первый e2e‑сценарий, где одноимённые задачи в двух секциях создают неоднозначность.
