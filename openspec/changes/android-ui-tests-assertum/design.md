## Context

DailyDo — приложение для трекинга задач на Compose Multiplatform. Экраны: My Lists (главный), Create New List, Progress View. Android таргет готов и запускается на эмуляторе. Assertum MCP Server предоставляет инструменты для записи и выполнения UI тестов через accessibility tree и screenshot-верификацию.

Текущее состояние: существуют требования к UI тестам в `openspec/specs/mobile-ui-tests/spec.md`, но конкретные тест-сценарии в Assertum не созданы. Нужно создать 5 E2E тестов, которые запускаются последовательно и сохраняются.

## Goals / Non-Goals

**Goals:**
- Создать 5 именованных E2E тестов через Assertum MCP Server
- Каждый тест записывается, верифицируется и сохраняется через `assertum_save_test`
- Тесты запускаются последовательно (предыдущий завершается → стартует следующий)
- Каждый тест независим: начинается с `assertum_restart_test` для чистого состояния
- Покрыть все 3 экрана и основные пользовательские сценарии

**Non-Goals:**
- iOS UI тесты (только Android)
- Интеграция с CI/CD pipeline
- Параллельный запуск тестов
- Тестирование backend API

## Decisions

### 1. Последовательность запуска тестов

**Решение**: Каждый тест запускается независимо через `assertum_restart_test` перед началом. Порядок:
1. `empty-state-create-list` — базовый флоу, создаёт данные для следующих тестов
2. `task-toggle-completion` — зависит от наличия списка
3. `navigation-flow` — чистая навигация, независима от данных
4. `create-list-multiple-tasks` — создание списка с 3+ задачами
5. `full-day-workflow` — полный рабочий день: создание + выполнение всех задач

**Почему не параллельно**: Assertum MCP работает с одним эмулятором, параллельность невозможна технически.

### 2. Инструменты Assertum для каждого шага

**Решение**: Использовать специализированные инструменты по назначению:
- `assertum_screenshot` — для визуальной верификации экранов
- `assertum_accessibility_tree` / `assertum_tree` — для поиска элементов по UI дереву
- `assertum_action` — для выполнения tap, input, scroll
- `assertum_assertion` — для проверки состояния элементов
- `assertum_save_test` — обязательный финальный шаг каждого теста

### 3. Структура каждого теста

**Решение**: Стандартная структура:
```
restart → screenshot (initial) → действия → assertion → screenshot (final) → save
```

**Почему**: restart гарантирует чистое состояние; финальный screenshot и assertion подтверждают результат; save сохраняет тест для повторного воспроизведения.

### 4. Именование тестов в Assertum

**Решение**: Использовать snake_case идентификаторы, соответствующие сценариям:
- `empty_state_create_list`
- `task_toggle_completion`  
- `navigation_flow`
- `create_list_multiple_tasks`
- `full_day_workflow`

## Risks / Trade-offs

- [Риск: Эмулятор не запущен] → Митигация: Перед запуском тестов проверить статус через `assertum_status`; инструкция в tasks.md по настройке adb forward
- [Риск: Нестабильные accessibility tree ID] → Митигация: Использовать текстовый поиск по содержимому, а не hardcoded ID
- [Риск: Тест 2 зависит от данных теста 1] → Митигация: Каждый тест создаёт нужные данные самостоятельно через restart + setup
- [Риск: Временные задержки анимации] → Митигация: Assertum MCP автоматически ждёт стабильности UI перед действием
